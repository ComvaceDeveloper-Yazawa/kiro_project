# ベースラインセキュリティルール

## 概要

これらのセキュリティルールは、すべてのAI-DLCフェーズに適用される**必須の**横断的制約です。これらはオプションのガイダンスではなく、質問の生成、設計成果物の作成、コード生成、完了メッセージの提示時にステージが**必ず**実施しなければならない厳格な制約です。

**実施**: 各該当ステージにおいて、モデルはステージ完了メッセージをユーザーに提示する前に、これらのルールへの準拠を検証しなければなりません。

### ブロッキングセキュリティ所見の動作

**ブロッキングセキュリティ所見**とは：

1. 所見は、ステージ完了メッセージの「セキュリティ所見」セクションにSECURITYルールIDと説明とともに記載されなければならない
2. すべてのブロッキング所見が解決されるまで、ステージは「次のステージに進む」オプションを提示してはならない
3. モデルは、何を変更する必要があるかを明確に説明した「変更を要求」オプションのみを提示しなければならない
4. 所見は、SECURITYルールID、説明、ステージコンテキストとともに`aidlc-docs/audit.md`に記録されなければならない

SECURITYルールが現在のプロジェクトに適用されない場合（例：データストアが存在しない場合のSECURITY-01）、コンプライアンスサマリーで**N/A**とマークしてください。これはブロッキング所見ではありません。

### デフォルトの実施

このドキュメントのすべてのルールは、デフォルトで**ブロッキング**です。いずれかのルールの検証基準が満たされない場合、それはブロッキングセキュリティ所見となります。上記で定義されたブロッキング所見の動作に従ってください。

### 検証基準の形式

このドキュメントの検証項目は、コンプライアンスチェックを説明する通常の箇条書きです。これらは、ステージ計画ファイルで使用される`- [ ]` / `- [x]`の進捗追跡チェックボックスとは異なります。各項目は、レビュー時に準拠または非準拠として評価されるべきです。

## ルール SECURITY-01: 保存時および転送時の暗号化

**ルール**: すべてのデータ永続化ストア（データベース、オブジェクトストレージ、ファイルシステム、キャッシュ、またはそれに相当するもの）は、以下を備えていなければなりません：

- マネージドキーサービスまたは顧客管理キーを使用した保存時の暗号化が有効
- 転送時の暗号化が強制されている（ストアへの出入りのすべてのデータ移動にTLS 1.2以上）

**検証**:

- 暗号化設定ブロックなしで定義されたストレージリソースがないこと
- 暗号化されていないプロトコルを使用するデータベース接続文字列がないこと
- オブジェクトストレージが保存時の暗号化を強制し、ポリシーによって非TLSリクエストを拒否すること
- データベースインスタンスでストレージ暗号化が有効で、TLS接続を強制していること

---

## ルール SECURITY-02: ネットワーク中継機器でのアクセスログ記録

**ルール**: 外部トラフィックを処理するすべてのネットワーク対応中継機器は、アクセスログを有効にしなければなりません。これには以下が含まれます：

- ロードバランサー → 永続ストアへのアクセスログ
- APIゲートウェイ → 集中ログサービスへの実行ログとアクセスログ
- CDNディストリビューション → 標準ログまたはリアルタイムログ

**検証**:

- アクセスログが有効になっていないロードバランサーリソースが定義されていないこと
- アクセスログが設定されていないAPIゲートウェイステージが定義されていないこと
- ログ設定なしで定義されたCDNディストリビューションがないこと

---

## ルール SECURITY-03: アプリケーションレベルのログ記録

**ルール**: デプロイされたすべてのアプリケーションコンポーネントは、構造化されたログ記録インフラストラクチャを含まなければなりません：

- ログ記録フレームワークが設定されていなければならない
- ログ出力は集中ログサービスに送信されなければならない
- ログには以下を含まなければならない：タイムスタンプ、相関/リクエストID、ログレベル、メッセージ
- 機密データ（パスワード、トークン、PII）はログ出力に表示されてはならない

**検証**:

- すべてのサービス/関数エントリポイントに設定されたロガーが含まれていること
- 本番コードの主要なログ記録メカニズムとしてアドホックなログステートメントが使用されていないこと
- ログ設定が集中ログサービスに出力をルーティングしていること
- シークレット、トークン、またはPIIがログに記録されていないこと

---

## ルール SECURITY-04: Webアプリケーションの HTTPセキュリティヘッダー

**ルール**: すべてのHTML配信エンドポイントで、以下のHTTPレスポンスヘッダーを設定しなければなりません：

| ヘッダー                    | 必須値                                                 |
| --------------------------- | ------------------------------------------------------ |
| `Content-Security-Policy`   | 制限的なポリシーを定義（最低限：`default-src 'self'`） |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains`                  |
| `X-Content-Type-Options`    | `nosniff`                                              |
| `X-Frame-Options`           | `DENY`（フレーミングが必要な場合は`SAMEORIGIN`）       |
| `Referrer-Policy`           | `strict-origin-when-cross-origin`                      |

**注**: `X-XSS-Protection`は最新のブラウザでは非推奨です。代わりに`Content-Security-Policy`を使用してください。

**検証**:

- ミドルウェアまたはレスポンスインターセプターがすべての必須ヘッダーを設定していること
- CSPポリシーが文書化された正当な理由なしに`unsafe-inline`または`unsafe-eval`を使用していないこと
- HSTSのmax-ageが少なくとも31536000（1年）であること

---

## ルール SECURITY-05: すべてのAPIパラメータの入力検証

**ルール**: すべてのAPIエンドポイント（REST、GraphQL、gRPC、WebSocket）は、処理前にすべての入力パラメータを検証しなければなりません。検証には以下を含める必要があります：

- **型チェック**: 予期しない型を拒否
- **長さ/サイズの境界**: 文字列の最大長、配列とペイロードの最大サイズを強制
- **形式検証**: 構造化された入力（メール、日付、ID）に許可リスト（正規表現またはスキーマ）を使用
- **サニタイゼーション**: XSSを防ぐため、ユーザー提供文字列内のHTML/スクリプトコンテンツをエスケープまたは拒否
- **インジェクション防止**: すべてのデータベース操作にパラメータ化クエリを使用（文字列連結は決して使用しない）

**検証**:

- すべてのAPIハンドラーが検証ライブラリまたはスキーマを使用していること
- 生のユーザー入力がSQL、NoSQL、またはOSコマンドに連結されていないこと
- 文字列入力に明示的な最大長制約があること
- リクエストボディサイズ制限がフレームワークまたはゲートウェイレベルで設定されていること

---

## ルール SECURITY-06: 最小権限アクセスポリシー

**ルール**: すべてのアイデンティティおよびアクセス管理ポリシー、ロール、または権限境界は、最小権限に従わなければなりません：

- 特定のリソース識別子を使用 — APIがリソースレベルの権限をサポートしていない場合を除き、ワイルドカードリソースは**決して**使用しない（例外を文書化）
- 特定のアクションを使用 — ワイルドカードアクションは**決して**使用しない
- 可能な限り条件をスコープする
- 読み取りと書き込みの権限を別々のポリシーステートメントに分離

**検証**:

- 文書化された例外なしに、ワイルドカードアクションまたはワイルドカードリソースを含むポリシーがないこと
- サービスロールが実際に呼び出すものよりも広い権限を持っていないこと
- 可能な限りマネージドポリシーを優先し、インラインポリシーを避けていること
- すべてのロールに特定のサービスまたはアカウントにスコープされた信頼ポリシーがあること

---

## ルール SECURITY-07: 制限的なネットワーク設定

**ルール**: すべてのネットワーク設定（セキュリティグループ、ネットワークACL、ルートテーブル）は、デフォルト拒否に従わなければなりません：

- ファイアウォールルール: アプリケーションに必要な特定のポートのみを開く
- 公開ロードバランサーのポート80/443を除き、ソース`0.0.0.0/0`のインバウンドルールはない
- 明示的に正当化されない限り、すべてのポートで`0.0.0.0/0`のアウトバウンドルールはない
- プライベートサブネットは直接インターネットゲートウェイルートを持ってはならない
- 利用可能な場合、クラウドサービスアクセスにプライベートエンドポイントを使用

**検証**:

- 公開ロードバランサーのポート80/443以外で、インバウンド`0.0.0.0/0`を許可するファイアウォールルールがないこと
- データベースとアプリケーションのファイアウォールルールが、ソースを特定のCIDRブロックまたはセキュリティグループ参照に制限していること
- プライベートサブネットがNATゲートウェイ経由でルーティングされている（インターネットゲートウェイではない）
- 高トラフィックのクラウドサービス呼び出しにプライベートエンドポイントが使用されていること

---

## ルール SECURITY-08: アプリケーションレベルのアクセス制御

**ルール**: リソースにアクセスまたは変更するすべてのアプリケーションエンドポイントは、アプリケーション層で認可チェックを実施しなければなりません：

- **デフォルト拒否**: すべてのルート/エンドポイントは、明示的に公開とマークされない限り、認証を要求しなければならない
- **オブジェクトレベルの認可**: IDでリソースを参照するすべてのリクエストは、リクエストしているユーザー/プリンシパルがそのリソースを所有しているか、アクセス権限を持っているかを検証しなければならない（IDORを防ぐ）
- **機能レベルの認可**: 管理者または特権操作は、呼び出し元のロール/権限をサーバー側でチェックしなければならない — クライアント側の非表示に決して依存しない
- **CORSポリシー**: クロスオリジンリソース共有は、明示的に許可されたオリジンに制限されなければならない — 認証されたエンドポイントで`Access-Control-Allow-Origin: *`を決して使用しない
- **トークン検証**: JWTまたはセッショントークンは、すべてのリクエストでサーバー側で検証されなければならない（署名、有効期限、オーディエンス、発行者）

**検証**:

- すべてのコントローラー/ハンドラーに認可ミドルウェアまたはガードが適用されていること
- 呼び出し元の所有権または権限を検証せずに、リソースIDのデータを返すエンドポイントがないこと
- 管理者/特権ルートに、サーバー側で実施される明示的なロールチェックがあること
- CORS設定が認証されたエンドポイントでワイルドカードオリジンを使用していないこと
- トークン検証がすべてのリクエストでサーバー側で行われている（ログイン時だけでなく）

---

## ルール SECURITY-09: セキュリティ強化と設定ミスの防止

**ルール**: デプロイされたすべてのコンポーネントは、強化ベースラインに従わなければなりません：

- **デフォルト認証情報なし**: デフォルトのユーザー名/パスワードは、デプロイ前に変更または無効化されなければならない
- **最小限のインストール**: 未使用の機能、フレームワーク、サンプルアプリケーション、ドキュメントエンドポイントを削除または無効化
- **エラー処理**: 本番環境のエラーレスポンスは、スタックトレース、内部パス、フレームワークバージョン、データベース詳細をエンドユーザーに公開してはならない
- **ディレクトリリスティング**: Webサーバーはディレクトリリスティングを無効にしなければならない
- **クラウドストレージ**: クラウドオブジェクトストレージは、明示的に必要で文書化されていない限り、パブリックアクセスをブロックしなければならない
- **パッチ管理**: ランタイム環境、フレームワーク、OSイメージは、現在サポートされているバージョンを使用しなければならない

**検証**:

- 設定ファイル、環境変数、IaCテンプレートにデフォルト認証情報が存在しないこと
- 本番環境のエラーレスポンスが一般的なメッセージを返す（スタックトレースや内部詳細なし）
- クラウドオブジェクトストレージが、文書化された例外が存在しない限り、パブリックアクセスをブロックしていること
- サンプル/デモアプリケーションやデフォルトページがデプロイされていないこと
- フレームワークとランタイムバージョンが現在サポートされていること

---

## ルール SECURITY-10: ソフトウェアサプライチェーンセキュリティ

**ルール**: すべてのプロジェクトは、ソフトウェアサプライチェーンを管理しなければなりません：

- **依存関係のピン留め**: すべての依存関係は、正確なバージョンまたはロックファイルを使用しなければならない
- **脆弱性スキャン**: 依存関係の脆弱性スキャナーが設定されていなければならない
- **未使用の依存関係なし**: 積極的に使用されていないパッケージを削除
- **信頼できるソースのみ**: 依存関係は、公式レジストリまたは検証済みのプライベートレジストリから取得しなければならない — 未検証のサードパーティソースは使用しない
- **SBOM**: プロジェクトは、本番デプロイメント用のソフトウェア部品表を生成しなければならない
- **CI/CD整合性**: ビルドパイプラインは、ピン留めされたツールバージョンと検証済みのベースイメージを使用しなければならない — 本番DockerfileまたはCI設定で`latest`タグを使用しない

**検証**:

- ロックファイルが存在し、バージョン管理にコミットされていること
- 依存関係の脆弱性スキャンステップがCI/CDに含まれているか、ビルド手順に文書化されていること
- 未使用または放棄された依存関係が含まれていないこと
- DockerfileとCI設定が本番環境で`latest`またはピン留めされていないイメージタグを使用していないこと
- 依存関係が公式または検証済みのレジストリから取得されていること

---

## ルール SECURITY-11: セキュアな設計原則

**ルール**: アプリケーション設計は、最初からセキュリティを組み込まなければなりません：

- **関心の分離**: セキュリティクリティカルなロジック（認証、認可、決済処理）は、専用モジュールに分離されなければならない — コードベース全体に散在させない
- **多層防御**: 単一の制御が唯一の防御線であってはならない — 制御を重ねる（検証 + 認可 + 暗号化）
- **レート制限**: 公開エンドポイントは、悪用を防ぐためにレート制限またはスロットリングを実装しなければならない
- **ビジネスロジックの悪用**: 設計は、ハッピーパスシナリオだけでなく、誤用ケースを考慮しなければならない

**検証**:

- セキュリティクリティカルなロジックが専用モジュールまたはサービスにカプセル化されていること
- 公開APIにレート制限が設定されていること
- 設計ドキュメントが少なくとも1つの誤用/悪用シナリオに対処していること

---

## ルール SECURITY-12: 認証と認証情報管理

**ルール**: ユーザー認証を持つすべてのアプリケーションは、以下を実装しなければなりません：

- **パスワードポリシー**: 最低8文字、侵害されたパスワードリストに対してチェック
- **認証情報の保存**: パスワードは適応型アルゴリズムを使用してハッシュ化されなければならない — 弱いまたは非適応型ハッシュは決して使用しない
- **多要素認証**: MFAは管理者アカウントでサポートされなければならず、すべてのユーザーで利用可能であるべき
- **セッション管理**: セッションはサーバー側で有効期限を持ち、ログアウト時に無効化され、secure/httpOnly/sameSite Cookie属性を使用しなければならない
- **ブルートフォース保護**: ログインエンドポイントは、繰り返し失敗後にアカウントロックアウト、段階的遅延、またはCAPTCHAを実装しなければならない
- **ハードコードされた認証情報なし**: ソースコードまたはIaCテンプレートにパスワード、APIキー、シークレットを含めない — シークレットマネージャーを使用

**検証**:

- パスワードハッシュが適応型アルゴリズムを使用している（弱いまたは非適応型ハッシュではない）
- セッションCookieが`Secure`、`HttpOnly`、`SameSite`属性を設定していること
- ログインエンドポイントにブルートフォース保護（ロックアウト、遅延、またはCAPTCHA）があること
- ソースコードまたは設定ファイルにハードコードされた認証情報がないこと
- 管理者アカウントでMFAがサポートされていること
- セッションがログアウト時に無効化され、定義された有効期限を持っていること

---

## ルール SECURITY-13: ソフトウェアとデータの整合性検証

**ルール**: システムは、ソフトウェアとデータの整合性を検証しなければなりません：

- **デシリアライゼーションの安全性**: 信頼できないデータは、検証なしにデシリアライズしてはならない — 安全なデシリアライゼーションライブラリまたは許可された型の許可リストを使用
- **アーティファクトの整合性**: ダウンロードされた依存関係、プラグイン、更新は、チェックサムまたはデジタル署名によって検証されなければならない
- **CI/CDパイプラインセキュリティ**: ビルドパイプラインは、パイプライン定義を変更できる人を制限しなければならない — コード作成者とデプロイ承認者の職務を分離
- **CDNと外部リソース**: 外部CDNからロードされるスクリプトまたはリソースは、サブリソース整合性（SRI）ハッシュを使用しなければならない
- **データ整合性**: 重要なデータ変更は監査可能でなければならない（誰が何をいつ変更したか）

**検証**:

- 信頼できない入力の安全でないデシリアライゼーションがないこと
- CDNからロードされる外部スクリプトにSRI整合性属性が含まれていること
- CI/CDパイプライン定義がアクセス制御され、変更が監査可能であること
- 重要なデータ変更が、アクター、タイムスタンプ、変更前後の値とともにログに記録されていること

---

## ルール SECURITY-14: アラートとモニタリング

**ルール**: ログ記録（SECURITY-02、SECURITY-03）に加えて、システムには以下を含めなければなりません：

- **セキュリティイベントアラート**: 高価値のセキュリティイベントに対してアラートを設定しなければならない：繰り返しの認証失敗、特権昇格の試み、異常な場所からのアクセス、認可失敗
- **ログの整合性**: ログは追加専用または改ざん防止ストレージに保存されなければならない — アプリケーションコードは自身の監査ログを削除または変更できてはならない
- **ログ保持**: ログは、アプリケーションのコンプライアンス要件に適した最小期間保持されなければならない（デフォルト：最低90日）
- **モニタリングダッシュボード**: 主要な運用およびセキュリティメトリクスのモニタリングダッシュボードまたはアラーム設定が定義されていなければならない

**検証**:

- 認証失敗と認可違反に対してアラートが設定されていること
- アプリケーションログ
  グループに保持ポリシーが設定されていること（最低90日）
- アプリケーションロールが自身のログ グループ/ストリームを削除する権限を持っていないこと
- セキュリティ関連イベント（ログイン失敗、アクセス拒否、特権変更）がアラートを生成すること

## ルール SECURITY-15: 例外処理とフェイルセーフデフォルト

**ルール**: すべてのアプリケーションは、例外的な状況を安全に処理しなければなりません：

- **キャッチと処理**: すべての外部呼び出し（データベース、API、ファイルI/O）は、明示的なエラー処理を持たなければならない — 本番環境で未処理のプロミス拒否または未キャッチ例外がないこと
- **フェイルクローズ**: エラー時、システムはアクセスを拒否するか、操作を停止しなければならない — 決してフェイルオープンしない
- **リソースクリーンアップ**: エラーパスはリソース（接続、ファイルハンドル、ロック）を解放しなければならない — try/finally、usingステートメント、または同等のパターンを使用
- **ユーザー向けエラー**: ユーザーに表示されるエラーメッセージは一般的でなければならない — 内部詳細やシステム情報なし
- **グローバルエラーハンドラー**: アプリケーションは、未処理の例外をキャッチし、ログに記録し（SECURITY-03に従って）、安全なレスポンスを返すグローバル/トップレベルエラーハンドラーを持たなければならない

**検証**:

- すべての外部呼び出し（DB、HTTP、ファイルI/O）に明示的なエラー処理がある（try/catch、.catch()、エラーコールバック）
- アプリケーションエントリポイントでグローバルエラーハンドラーが設定されていること
- エラーパスが認可または検証チェックをバイパスしない（フェイルクローズ）
- エラーパスでリソースがクリーンアップされている（接続が閉じられ、トランザクションがロールバックされる）
- アプリケーションコードに未処理のプロミス拒否または未キャッチ例外の警告がないこと

## 実施の統合

これらのルールは、すべてのAI-DLCステージに適用される横断的制約です。各ステージで：

- 生成された成果物に対してすべてのSECURITYルール検証基準を評価
- ステージ完了サマリーに「セキュリティコンプライアンス」セクションを含め、各ルールを準拠、非準拠、またはN/Aとしてリスト
- いずれかのルールが非準拠の場合、これはブロッキングセキュリティ所見です — 概要で定義されたブロッキング所見の動作に従ってください
- 設計ドキュメントとテスト手順にセキュリティルール参照を含める
