# ビジネスルール - Unit 3: backend

## ルール定義の形式

各ルールは「推奨 / 非推奨 / 理由」形式で定義します。

---

## 1. レイヤー間依存ルール

| 推奨                                               | 非推奨                                               | 理由                                              |
| -------------------------------------------------- | ---------------------------------------------------- | ------------------------------------------------- |
| Interface → Application → Domain の一方向依存      | Domain 層が Infrastructure 層を直接 import           | 依存方向が逆転するとテスタビリティが失われる      |
| Infrastructure 層は Domain のリポジトリ I/F を実装 | Infrastructure 層のクラスを Application 層で直接 new | DI が崩れ、テスト時にモック差し替えができなくなる |
| ユースケースはリポジトリ I/F（抽象）に依存         | ユースケースが Prisma クライアントを直接使用         | Prisma の変更がユースケースに波及する             |

---

## 2. Fastify プラグイン登録ルール

| 推奨                                                             | 非推奨                                             | 理由                                                             |
| ---------------------------------------------------------------- | -------------------------------------------------- | ---------------------------------------------------------------- |
| セキュリティ系プラグイン（cors・helmet・rate-limit）を最初に登録 | ルートプラグインの後にセキュリティプラグインを登録 | 一部のリクエストにセキュリティヘッダーが適用されない可能性がある |
| `db.plugin` を `auth.plugin` より先に登録                        | auth.plugin を db.plugin より先に登録              | auth.plugin が Prisma を使う場合に未初期化エラーが発生する       |
| `setErrorHandler` を全プラグイン登録後に設定                     | プラグイン登録前に setErrorHandler を設定          | 一部のプラグインエラーがハンドラーに到達しない                   |

---

## 3. JWT 検証ルール

| 推奨                                                        | 非推奨                                     | 理由                                           |
| ----------------------------------------------------------- | ------------------------------------------ | ---------------------------------------------- |
| `supabase.auth.getUser(token)` で検証                       | トークンをデコードのみして検証をスキップ   | 署名検証なしでは偽造トークンを受け入れてしまう |
| 全認証必須ルートに `preHandler` フックで auth.plugin を適用 | ルートハンドラー内で個別に JWT 検証        | 検証漏れが発生しやすい                         |
| 認証失敗時は `UnauthorizedError` を throw                   | 認証失敗時に null を返してハンドラーで分岐 | エラーハンドリングが分散し、一貫性が失われる   |
| `request.user` に検証済みユーザー情報を付与                 | ハンドラー内でトークンを再検証             | 二重検証は不要なオーバーヘッドになる           |

---

## 4. Prisma クライアント管理ルール

| 推奨                                                    | 非推奨                                | 理由                                               |
| ------------------------------------------------------- | ------------------------------------- | -------------------------------------------------- |
| `fastify.decorate('prisma', client)` でシングルトン管理 | リクエストごとに `new PrismaClient()` | コネクションプールが枯渇する                       |
| `onClose` フックで `prisma.$disconnect()` を呼ぶ        | アプリ終了時に切断処理なし            | コネクションリークが発生する                       |
| `infrastructure/prisma/client.ts` でシングルトンを定義  | 各ファイルで `new PrismaClient()`     | 複数インスタンスが生成されコネクション数が増大する |

---

## 5. AppError 使用ルール

| 推奨                                                             | 非推奨                                                  | 理由                                            |
| ---------------------------------------------------------------- | ------------------------------------------------------- | ----------------------------------------------- |
| ドメイン層・ユースケース層では AppError サブクラスを throw       | `reply.status(404).send(...)` を直接呼ぶ                | Interface 層の詳細がドメイン層に漏れる          |
| `ServiceUnavailableError` は外部サービス障害時のみ使用           | 全ての予期しないエラーに ServiceUnavailableError を使用 | エラーの意味が曖昧になる                        |
| `InternalServerError` は予期しない例外のフォールバックとして使用 | catch ブロックで素の Error を throw                     | setErrorHandler が 500 として適切に処理できない |
| エラーメッセージはユーザー向けの日本語で記述                     | スタックトレースや内部パスをメッセージに含める          | SECURITY-09 違反（内部情報の漏洩）              |

---

## 6. ルート定義ルール

| 推奨                                                                   | 非推奨                                   | 理由                                                      |
| ---------------------------------------------------------------------- | ---------------------------------------- | --------------------------------------------------------- |
| `schema.safeParse()` で入力検証し、失敗時は `ValidationError` を throw | `schema.parse()` を try/catch で囲む     | safeParse の方が意図が明確で Zod エラーを適切に変換できる |
| Fastify の `schema` オプションに Zod スキーマを登録                    | ハンドラー内でのみバリデーション         | Fastify のシリアライズ最適化が効かなくなる                |
| ルートハンドラーはユースケースを呼ぶだけにする                         | ルートハンドラーにビジネスロジックを書く | Interface 層にビジネスロジックが漏れる                    |
| レスポンスは必ず `ApiResponse<T>` 形式で返す                           | 生のオブジェクトをそのまま返す           | フロントエンドとの型契約が崩れる                          |

---

## 7. リポジトリ実装ルール

| 推奨                                              | 非推奨                                    | 理由                                                                           |
| ------------------------------------------------- | ----------------------------------------- | ------------------------------------------------------------------------------ |
| Prisma の結果をドメインエンティティに変換して返す | Prisma の型をそのままドメイン層に返す     | Infrastructure の型がドメイン層に漏れる                                        |
| `findById` で存在しない場合は `null` を返す       | 存在しない場合に `NotFoundError` を throw | リポジトリはデータアクセスのみ担当し、ビジネスルールはユースケース層で判断する |
| Prisma クエリは `fastify.prisma` 経由で実行       | `new PrismaClient()` をリポジトリ内で生成 | シングルトン管理が崩れる                                                       |

---

## 8. セキュリティルール（SECURITY-01〜15 対応）

| ルール      | 適用内容                                                                                         |
| ----------- | ------------------------------------------------------------------------------------------------ |
| SECURITY-03 | 全エンドポイントで構造化ログ（timestamp・requestId・level・message）を出力                       |
| SECURITY-04 | `@fastify/helmet` で CSP・HSTS・X-Frame-Options 等を設定                                         |
| SECURITY-05 | 全ルートで Zod スキーマによる入力バリデーション必須                                              |
| SECURITY-08 | 全認証必須ルートに `preHandler` で auth.plugin を適用。IDOR 防止のためリソース所有者チェック必須 |
| SECURITY-09 | エラーレスポンスにスタックトレース・内部パスを含めない                                           |
| SECURITY-11 | `@fastify/rate-limit` でレート制限。認証エンドポイントは特に厳格に設定                           |
| SECURITY-12 | Supabase Auth に認証を委譲。ハードコードされた認証情報を禁止                                     |
| SECURITY-15 | `setErrorHandler` でグローバルエラーハンドラーを設定。全外部呼び出しに try/catch を適用          |

---

## 9. 環境変数管理ルール

| 推奨                                                  | 非推奨                         | 理由                                        |
| ----------------------------------------------------- | ------------------------------ | ------------------------------------------- |
| `.env.example` に全環境変数のキーとコメントを記載     | `.env` をリポジトリにコミット  | SECURITY-12 違反（クレデンシャルの漏洩）    |
| 環境変数は起動時に検証（未設定なら即時エラー）        | 環境変数未設定のまま起動を許可 | 実行時に予期しないエラーが発生する          |
| `DATABASE_URL` はマイグレーション時と実行時で切り替え | 常に Direct URL を使用         | Pooler を使わないとコネクション数が増大する |
