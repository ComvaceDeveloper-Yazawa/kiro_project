# ビジネスルール - Unit 4: frontend

## ルール定義の形式

各ルールは「推奨 / 非推奨 / 理由」形式で定義します。

---

## 1. 責務分離ルール（components / composables / stores）

| 推奨                                                  | 非推奨                                          | 理由                                           |
| ----------------------------------------------------- | ----------------------------------------------- | ---------------------------------------------- |
| ビジネスロジック・バリデーションは composables に実装 | `<script setup>` 内に直接ビジネスロジックを書く | コンポーネントが肥大化し、テストが困難になる   |
| API 通信は `composables/api/` に集約                  | コンポーネント内で直接 `fetch` を呼ぶ           | 認証ヘッダー付与・エラーハンドリングが分散する |
| stores は状態の保持のみ（ビジネスロジックを含めない） | store の action に複雑なビジネスロジックを書く  | composables との責務が混在する                 |
| `components/` は UI 描画とイベント受付のみ            | components に `ref` で状態を持ちすぎる          | 状態管理が分散し、テストが困難になる           |

---

## 2. Vue 3 script setup ルール

| 推奨                                                 | 非推奨                                           | 理由                                            |
| ---------------------------------------------------- | ------------------------------------------------ | ----------------------------------------------- |
| `<script setup lang="ts">` を使用                    | Options API / `defineComponent`                  | script setup の方が型推論が強力で記述量が少ない |
| props は `defineProps<{...}>()` でジェネリック型定義 | `defineProps({ prop: String })` のランタイム定義 | TypeScript の型チェックが効かない               |
| emits は `defineEmits<{...}>()` でジェネリック型定義 | `defineEmits(['update'])` の文字列配列           | 型安全性が失われる                              |
| `computed` で派生状態を定義                          | template 内に複雑な式を書く                      | テスタビリティと可読性が下がる                  |
| `watchEffect` より `watch` を優先（依存関係を明示）  | 副作用を `watchEffect` に任せる                  | 依存関係が暗黙的になりデバッグが困難            |

---

## 3. Pinia Setup 記法ルール

| 推奨                                             | 非推奨                                                    | 理由                                         |
| ------------------------------------------------ | --------------------------------------------------------- | -------------------------------------------- |
| `defineStore('id', () => { ... })` の Setup 記法 | Options 記法（`state`, `getters`, `actions`）             | Setup 記法の方が TypeScript との親和性が高い |
| store ID はケバブケース（`'auth-store'`）        | スペースや大文字を含む ID                                 | Pinia DevTools での識別が困難になる          |
| `storeToRefs()` で state を分割代入              | store を直接分割代入（`const { user } = useAuthStore()`） | リアクティビティが失われる                   |
| store はコンポーネント・composables から使用     | store が他の store を直接 import して循環依存             | 循環依存でランタイムエラーが発生する         |

---

## 4. SCSS ルール（SMACSS ベース）

| 推奨                                                          | 非推奨                                                 | 理由                                         |
| ------------------------------------------------------------- | ------------------------------------------------------ | -------------------------------------------- |
| レイアウトクラスに `l-` プレフィックス（`l-container`）       | レイアウトとモジュールを同じクラスに混在               | SMACSS の責務分離が崩れる                    |
| 状態クラスに `is-` プレフィックス（`is-active`, `is-hidden`） | 状態を直接スタイルで表現（`display: none` を直書き）   | 状態管理が分散する                           |
| コンポーネント内 SCSS は `<style scoped lang="scss">`         | グローバルスタイルにコンポーネント固有のスタイルを書く | スタイルの衝突が発生する                     |
| `$space-*` 変数でスペーシングを統一                           | マジックナンバー（`margin: 24px`）を直書き             | デザインの一貫性が失われる                   |
| `$color-*` 変数でカラーを統一                                 | カラーコードを直書き（`color: #1d1d1f`）               | テーマ変更時に全ファイルを修正する必要がある |
| Apple スタイル: 余白は大きめに（`$space-12` 以上）            | 要素を詰め込む                                         | Apple.com の「呼吸する」デザインが失われる   |

---

## 5. Vue Router ルール

| 推奨                                                        | 非推奨                                     | 理由                                   |
| ----------------------------------------------------------- | ------------------------------------------ | -------------------------------------- |
| 認証必須ルートに `meta: { requiresAuth: true }` を設定      | ナビゲーションガード内で個別に認証チェック | 設定漏れが発生しやすい                 |
| ページコンポーネントは遅延ロード（`() => import(...)`）     | 全コンポーネントを静的 import              | 初期バンドルサイズが増大する           |
| ルートファイルは機能ごとに分割（`auth.routes.ts`）          | 全ルートを `routes.ts` に列挙              | ルートが増えると管理が困難になる       |
| `router.push({ name: 'RouteName' })` で名前付きルートを使用 | `router.push('/path')` でパスを直書き      | パス変更時に全箇所を修正する必要がある |

---

## 6. API クライアントルール

| 推奨                                                 | 非推奨                                     | 理由                                       |
| ---------------------------------------------------- | ------------------------------------------ | ------------------------------------------ |
| `useAuthStore` からトークンを取得                    | `localStorage.getItem('token')` を直接呼ぶ | 型安全性がなく、トークン更新が反映されない |
| `ApiResponse<T>` 型で受け取り `success` フラグで分岐 | レスポンスを `any` で受け取る              | 型安全性が失われる                         |
| API エラーは composables 層でハンドリング            | コンポーネント内で try/catch               | エラーハンドリングが分散する               |
| ベース URL は環境変数（`VITE_API_BASE_URL`）で管理   | URL をハードコード                         | 環境ごとの切り替えができない               |

---

## 7. フォームバリデーションルール

| 推奨                                                                            | 非推奨                             | 理由                                           |
| ------------------------------------------------------------------------------- | ---------------------------------- | ---------------------------------------------- |
| `watch(formData, ..., { deep: true })` でリアルタイム検証                       | submit 時のみ検証                  | ユーザーが送信後に初めてエラーを知る UX になる |
| Zod の `safeParse().error.flatten().fieldErrors` でフィールドごとのエラーを取得 | 手動でバリデーションロジックを書く | Zod スキーマと二重管理になる                   |
| エラーメッセージは日本語（shared の Zod スキーマに準拠）                        | 英語エラーメッセージ               | ユーザー向けアプリは日本語で統一               |
| エラー表示は入力フィールドの直下に小さく表示                                    | モーダルやアラートでエラーを表示   | Apple スタイルのインラインエラー表示           |

---

## 8. セキュリティルール（SECURITY-04 / SECURITY-08 対応）

| ルール      | 適用内容                                                                       |
| ----------- | ------------------------------------------------------------------------------ |
| SECURITY-04 | CSP は backend の helmet で設定。frontend は `unsafe-inline` を使わない        |
| SECURITY-08 | ナビゲーションガードで認証チェック。`meta.requiresAuth` 未設定ルートは公開扱い |
| SECURITY-09 | API エラーの詳細（スタックトレース等）をユーザーに表示しない                   |
| SECURITY-12 | トークンは `useAuthStore` 経由で管理。`localStorage` への直接アクセス禁止      |

---

## 9. アクセシビリティルール（data-testid）

| 推奨                                                                     | 非推奨                                 | 理由                               |
| ------------------------------------------------------------------------ | -------------------------------------- | ---------------------------------- |
| インタラクティブ要素に `data-testid` を付与                              | テスト時に CSS クラスや DOM 構造に依存 | リファクタリング時にテストが壊れる |
| 命名規則: `{component}-{element-role}`（例: `login-form-submit-button`） | 連番や自動生成 ID                      | テストの意図が不明確になる         |
| `aria-label` で操作の意味を明示                                          | アイコンのみのボタンに説明なし         | スクリーンリーダーで操作できない   |
