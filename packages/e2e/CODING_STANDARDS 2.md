# E2E / CI-CD コーディング規約

## E2E テスト方針（Playwright）

### テスト対象の選定基準

E2E テストは「主要フローのみ」に絞る。以下を対象とする:

- 認証フロー（ログイン・ログアウト）
- コアユーザーフロー（アプリの主要な操作）
- 回帰リスクの高い機能

以下は対象外（Unit / Integration テストで担保）:

- 個別コンポーネントの動作
- API レスポンスの検証
- エラーメッセージの文言

---

## 1. セレクター規則

| 推奨                                           | 非推奨                         | 理由                           |
| ---------------------------------------------- | ------------------------------ | ------------------------------ |
| `page.getByTestId('login-form-submit-button')` | `page.locator('.btn-primary')` | CSS クラス変更でテストが壊れる |
| `data-testid` 属性を使用                       | `id` や `class` セレクター     | リファクタリング耐性が高い     |
| `{component}-{element-role}` 命名              | 連番や自動生成 ID              | テストの意図が明確になる       |

---

## 2. テスト設計ルール

| 推奨                                         | 非推奨                                     |
| -------------------------------------------- | ------------------------------------------ |
| 各テストは独立して実行できる                 | 前のテストの状態に依存する                 |
| `test.beforeEach` でテスト前の状態をリセット | グローバル状態を共有する                   |
| `expect` は具体的な値を検証                  | `expect(element).toBeTruthy()` のみ        |
| テストデータは環境変数で管理                 | テストコードにクレデンシャルをハードコード |

---

## 3. CI/CD ワークフロー変更ルール

- ワークフローファイルの変更は PR レビュー必須
- シークレット（`secrets.*`）はリポジトリの Settings > Secrets で管理
- `latest` タグの Action は使用しない（バージョンを固定する）
- デプロイステップは E2E テスト成功後にのみ実行する（`needs: e2e`）

---

## 4. 必要な GitHub Secrets

| シークレット名           | 用途                                  |
| ------------------------ | ------------------------------------- |
| `VITE_API_BASE_URL`      | frontend ビルド時の API URL           |
| `VITE_SUPABASE_URL`      | frontend ビルド時の Supabase URL      |
| `VITE_SUPABASE_ANON_KEY` | frontend ビルド時の Supabase Anon Key |
| `STAGING_URL`            | E2E テストの対象 URL                  |

---

## 5. 新しい E2E テストを追加する手順

1. `packages/e2e/tests/{feature}.spec.ts` にテストファイルを作成
2. `data-testid` セレクターを使用（コンポーネント側に付与されていることを確認）
3. テストは独立して実行できることを確認（`pnpm --filter @monorepo/e2e test` でローカル実行）
4. CI で自動実行されることを確認
